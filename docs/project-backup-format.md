# `.beadproject` 备份协议

## 版本策略

- 当前导出版本：`2`
- 可导入版本：`1`、`2`
- 文件扩展名：`.beadproject`
- 外层格式标识：`bead-pattern-studio`

未知版本必须拒绝，不能猜测字段含义。v1 保留为只读兼容入口；重新导出时统一写成 v2。

## v2 外层结构

```json
{
  "format": "bead-pattern-studio",
  "version": 2,
  "exportedAt": "2026-07-19T00:00:00.000Z",
  "projects": []
}
```

每个项目保留标题、来源名称、保存时间、分类、复刻来源、设置和缩略图。色卡只保存 `code`、`name`、`hex`；`rgb` 和 `lab` 在导入时由 HEX 重新计算，避免信任可篡改的派生值。

## 紧凑图纸结构

```json
{
  "width": 48,
  "height": 48,
  "colors": [
    { "code": "H7", "hex": "#000000" }
  ],
  "cellColors": "...base64...",
  "sourceRgb": "...base64..."
}
```

- `colors` 是该图纸实际使用的色号字典，同一色号只能对应一个 HEX。
- `cellColors` 按行优先保存每格色号字典索引，每格为小端序 `Uint16`，正好两个字节。
- `sourceRgb` 按行优先保存每格源色，每格依次为 R、G、B 三个 8 位字节。
- 导入后格子数量必须严格等于 `width * height`。
- 每个色号索引必须指向 `colors` 中存在的项目。

## 校验边界

- 最多 100 个作品。
- 图纸单边最多 500 格。
- 色卡和图纸颜色字典最多 1000 色。
- 备份文件最多 96 MB。
- Base64 文本长度必须与预期字节数完全一致，解码后再次核对长度。
- 有正式色卡的项目中，格子色号和 HEX 必须与色卡一致。
- 项目 ID 在同一备份中必须唯一。

## 云同步映射

v2 是文件交换协议，不直接等于未来数据库表。云同步时建议：

- 项目元数据单独保存，并增加服务端 `revision`。
- 紧凑图纸正文作为不可变版本对象保存。
- 更新项目时提交上一个 `revision`，冲突时保留双方版本。
- 公开社区作品同时固定色卡版本 ID，不能只依赖当前默认色卡。
